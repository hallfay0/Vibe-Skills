# Vibe 路由 V2 测试框架正式方案

## 目的

这份文章把新一代 `vibe` 路由测试方案正式固定下来。

它要解决的不是“某一道题现在有没有路由对”，而是更大的问题：怎样用一套稳定、可迁移、不过拟合的测试框架，去保护下一代路由系统的通用质量。

这套正式方案服务于下面这一种新路由形态：

1. `SKILL.md` 仍然是唯一事实来源。
2. 路由证据直接从原始 `SKILL.md` 中提取，而不是依赖人工维护的技能卡。
3. 用户任务先被拆成几个能力模块。
4. 每个模块各自去检索候选证据和候选 skills。
5. 最终选择不再由一个“全局最高分”拍板，而是由覆盖度、互补性和合同约束共同决定。

这份测试方案要保护的是这套结构本身，而不是某一批现成提示词。

## 为什么旧测试形态不够

前面的 chaos probe 很有价值，因为它真实暴露了长期存在的路由病灶：

- 不相关的 skills 混进 shortlist。
- 复杂任务缺模块，最后只剩一个单点 skill。
- 很弱的文字碰撞被当成能力证据。
- 大而泛的 workflow skill 挤掉更直接的 owner skill。

但它仍然是下游测试。它只能告诉我们“这次路由看起来不对”，却很难回答“到底是哪一层坏了”。

如果下一代路由系统仍然只靠这种黑箱结果测试，我们就会不断回到旧路上：看到某个题面失败，就去补一个局部 patch。这样做短期能止血，长期一定会把系统重新推回场景适配。

所以，新测试框架必须把不同结构层分开测试，至少区分清楚：

- 文档证据是怎么被抽出来的；
- 任务模块是怎么被识别出来的；
- 候选是怎么被召回的；
- 最终多 skill 组合是怎么形成的；
- 对用户公开的话术是否忠实反映了真实路由结果。

## 正式设计原则

这套框架的核心原则只有一句话：

**测结构性不变量，不测个人偏爱的提示词结果。**

这里要保护的关键不变量有五个：

1. 写法变了，路由不应该大幅漂移。
2. 大段背景说明不应该压过直接用途证据。
3. 复合任务必须显出多个模块，不能塌缩成一个 winner。
4. 缺失能力时要诚实暴露缺口，不能假装覆盖到了。
5. `explicit_only`、`not_for` 这类明确合同必须一路保留下来。

这五条不是某个本地技能库的临时规则，而是跨环境都成立的质量要求。不管用户本地装的是科研 skills、视频剪辑 skills、前后端开发 skills，还是别的东西，这五条都应该成立。

## 五个公开测试缝

新框架固定五条公开测试缝。后续实现和重构都应该围绕这五条缝来立测试，而不是直接拿整条黑箱路由去猜内部问题。

### 一、证据分块缝

输入：

- 一个 `SKILL.md`

输出：

- 一组有顺序的证据块
- 每个证据块对应的原文位置
- 每个证据块的初步角色标签

这条缝保护的事情是：

- 系统能从真实、脏乱、写法不统一的 `SKILL.md` 里抽出可路由的证据视图；
- 代码块、教程段落、参考转储不会天然被当成强用途证据；
- 每条证据都能回溯到原文。

这一层该测什么：

- 分块边界是否稳定；
- 代码 fence、原始参考材料是否被降权或排除出强证据；
- 简短用途说明、用法 bullet、短说明段落是否仍可见；
- 原文 span 是否保留。

这一层不该去测精确排名，也不该绑定具体向量值。它只负责确认：原始文档有没有被转换成一个合理的“可供路由使用的证据视图”。

### 二、证据角色判定缝

输入：

- 一个证据块

输出：

- 角色标签
- 置信度

角色空间保持小而明确，当前正式定义为：

- `applicable`
- `not_applicable`
- `deliverable`
- `prerequisite`
- `example`
- `background`

这条缝保护的事情是：

- 系统能区分“这个 skill 适合做什么”和“这里举了一个关于什么的例子”；
- “不要拿它做什么”这种反向边界能保留下来；
- 背景知识不会偷偷变成路由权威。

这一层该测什么：

- 直接用途描述不能被误判成背景；
- 示例和教程段落不能被抬成强用途证据；
- 明确负边界必须保持为负边界；
- 弱证据的置信度必须低于直接用途证据。

### 三、任务模块抽取缝

输入：

- 一条用户任务

输出：

- 一小组任务模块

这里的“模块”指的是能力大小合适的工作片段，不是时间线步骤，也不是最终执行计划。例如：

- 文献检索
- 统计比较
- 中文改写
- 汇报整理
- 前端构建
- preview 部署
- 测试报告打包

这条缝保护的事情是：

- 复合任务会被识别成多模块问题；
- 简单任务不会被过度拆碎；
- 换一种说法，不应该完全换一张模块图。

这一层该测什么：

- 同义改写后的任务是否得到相近的模块集合；
- 复合任务是否产出多个模块；
- 单一小任务是否保持紧凑；
- 输出模块是否足够简洁，能进入后续检索。

### 四、模块候选召回缝

输入：

- 一个任务模块
- 一个本地 skill 语料库

输出：

- top-k 候选证据块
- top-k 候选 skills

这条缝保护的事情是：

- 面对陌生、本地化、用户自定义的 skill 库，系统仍能先把可能相关的候选找出来；
- 召回要够宽，保证 recall；
- 召回又不能太散，避免把噪声全抬上来。

这一层该测什么：

- top-k 里至少要出现一个正确家族；
- 又长又泛的 skill 不能只因为沾边词汇多就长期霸榜；
- `explicit_only` skill 不能作为默认候选乱入；
- skill 写法变化不能把召回直接打没。

这一层主要看 recall 和 leakage，不看某个固定分数。

### 五、最终组合缝

输入：

- 一组任务模块
- 每个模块的候选集合
- 路由等级，例如 `L` 或 `XL`

输出：

- 最终选中的 skill 集合
- 每个 skill 的入选理由
- 明确未覆盖模块

这条缝保护的事情是：

- 系统最终选的是一组能协作完成任务的 skills，而不是一个纯语义冠军；
- `L` 方案保持紧凑；
- `XL` 方案允许更完整的覆盖；
- 缺失模块要被诚实报出来，而不是由一个不太相关的 skill 冒充。

这一层该测什么：

- 多模块任务是否能选出互补组合；
- 同一表面的重复 skill 不能因为都高分就被一起塞满；
- `L` 和 `XL` 的差异应该来自组合策略，而不是措辞漂移；
- 未覆盖模块是否被明确说出来。

## 四层测试数据

正式方案把测试数据分成四层。不同层负责不同问题，不应该混成一个大杂烩 gate。

### A 层：最小合成 skill 库

这是默认回归的主骨架。

每个小语料库大约放 6 到 15 个 skills，故意包含：

- 1 到 2 个直接 owner；
- 1 个相邻但错误的 owner；
- 1 个宽泛 workflow skill；
- 1 个 `explicit_only` skill；
- 1 个噪声很长的文档型 skill；
- 在需要时，再故意留一个能力缺口。

这一层的目的：

- 保持测试快；
- 保持失败可解释；
- 不依赖当前这台机器的真实本地技能库。

### B 层：写法扰动语料

这一层用同一批小语料库，重写成多种风格：

- 极短说明型；
- 长教程型；
- bullet 密集型；
- 中英混写型；
- 几乎没有标题型；
- 示例堆叠型。

这一层的目的：

- 检验系统是不是过度依赖文档写法，而不是文档真实用途。

预期结果不是“完全一模一样”，而是家族级召回和最终组合要大体稳定。

### C 层：对抗噪声语料

这一层故意制造最容易出事故的语料：

- 很长的背景段；
- 夹带许多相邻领域词汇的示例；
- 语义上很像、但真实用途并不对应的教程；
- 明确的负边界；
- 很容易挤掉 owner 的大而泛 delivery skill。

这一层的目的：

- 测噪声抵抗力；
- 测合同约束是否还活着；
- 测系统会不会把背景知识抬成路由权威。

### D 层：真实宿主审计

这一层直接跑用户本地真实 skill 根。

这一层的目的：

- 观察真实漂移；
- 暴露脏文档和历史遗留格式；
- 发现合成语料里没有覆盖到的新失败类型。

但这一层必须明确降级为**可选审计**，不能并入默认 gate。原因很简单：真实本地 skill 库天然是用户私有、持续变化、写法不整齐的环境，拿它做默认回归只会让 gate 越来越脏、越来越难解释。

## 核心判定指标

新框架不应该强依赖“必须精确命中某个 skill 名”。正式判定指标改为下面六类：

### 1. 家族召回

对每个任务模块来说，top-k 里是否至少出现一个正确家族的候选。

### 2. 偏航泄漏

公开 shortlist 或最终集合里，出现了多少明显错误家族的 skills。

### 3. 覆盖完整度

对复合任务来说，最终选中的 skill 集合实际覆盖了多少必要模块。

### 4. 风格稳定性

同一批 skills 改写成不同文档风格后，召回和最终组合发生了多大变化。

### 5. 合同服从度

`explicit_only`、`not_for` 等合同约束是否贯穿整个路由过程。

### 6. 诚实缺口报告

当本地 skill 库确实缺某个模块时，系统有没有明确报缺，而不是编造一个看上去像的替代品。

这些指标比“skill X 必须 rank 1”更适合做通用框架，因为它们能跨不同用户的本地 skill 库迁移。

## 默认回归与可选审计的边界

正式方案要求把默认回归集压小、压实，只保结构性测试。

默认回归集保留：

- 证据分块合同测试；
- 证据角色判定合同测试；
- 合成任务上的模块抽取测试；
- 合成语料和扰动语料上的候选召回测试；
- 合成多模块任务上的最终组合测试；
- 少量高价值的对抗噪声测试。

默认回归集不保留、统一降级为可选审计：

- 跑完整 `.agents` 本地语料库的路由；
- 大型 prompt replay 包；
- live embedding 模型对比；
- live subagent 黑箱审计；
- 跨宿主环境对比。

这条边界很重要。默认回归负责保护核心能力；可选审计负责探索真实世界的脏面和新问题。两者都需要，但绝不能混在一个默认 gate 里。

## 确定性策略

默认回归不能依赖 live 模型调用。

正式要求是：默认回归尽可能使用确定性替身，包括：

- 固定证据块夹具；
- 固定角色标签夹具；
- 固定模块夹具；
- 确定性的检索或近邻测试替身。

只有可选审计层才允许接入 live 的抽取或检索能力。

这样做有两个直接好处：

1. 默认测试一旦失败，说明是路由合同被破坏，而不是模型今天状态漂了。
2. 本地迭代速度能保住，不会每改一次路由都要绑上整套在线语义基础设施。

## 夹具结构建议

为了让这套框架长期可维护，夹具结构也要固定下来。

### Skill 语料夹具

每个测试语料建议显式声明：

- skill id；
- 原始 `SKILL.md`；
- 预期能力家族；
- 合同标记，例如 `explicit_only`；
- 已知噪声面。

### Task 夹具

每个任务夹具建议显式声明：

- 任务原文；
- 预期模块；
- 预期家族覆盖；
- `L` / `XL` 下允许的紧凑度；
- 已知坏漂移样例。

### 组合夹具

每个组合测试夹具建议显式声明：

- 模块集合；
- 每个模块的候选集合；
- 预期最终家族覆盖；
- 最大允许冗余选择数；
- 在需要时，预期缺口报告。

## 场景家族覆盖面

正式方案不再围绕几道个人熟悉的 prompt 建体系，而是至少覆盖下面八类场景家族：

1. 单通道、明确请求；
2. 研究加写作的复合任务；
3. 软件调试和回归排查；
4. 构建、部署、验证三段式任务；
5. 改写、人类化和领域分析混合任务；
6. 媒体或内容生产复合任务；
7. 架构或方案拆解型任务；
8. 故意缺模块的任务。

这里固定的不是 skill 名，而是任务家族和应有的路由行为。

## 真实宿主审计协议

真实宿主审计仍然重要，但必须和默认回归彻底分开。

一次 live audit 至少做六件事：

1. 用真实本地 skill 语料跑完整路由流程；
2. 记录任务模块；
3. 记录每个模块的 top-k 候选；
4. 记录最终选中的 skill 集合；
5. 记录未覆盖模块；
6. 在需要时，用干净上下文 subagent 检查对用户公开的话术。

这层审计沿用同一套结构性指标：

- 模块召回；
- 偏航泄漏；
- 覆盖完整度；
- 合同服从度；
- 诚实缺口报告。

这样做的好处是：真实环境审计仍然被纳入统一框架，而不会重新退化成一次次临时 demo。

## TDD 落地顺序

正式落地顺序固定如下：

1. 证据分块缝；
2. 证据角色判定缝；
3. 任务模块抽取缝；
4. 模块候选召回缝；
5. 最终组合缝；
6. 真实宿主审计适配器。

这个顺序不能倒过来。

如果一开始就从 live route probe 直接打高层黑箱，我们几乎必然会再次掉回“看到哪个题错了就补哪个题”的局部修修补补。只有先把底层缝面立稳，上层路由行为才会变得可解释、可治理、可迭代。

## 成功标准

一项路由改动是不是好改动，正式判断标准应该是下面这些问题能不能被清楚回答：

- 文档证据变干净了还是更脏了？
- 模块召回提升了还是退步了？
- 最终覆盖更完整了还是更残缺了？
- 对写法差异更稳了还是更敏感了？
- 合同约束保住了还是漏掉了？
- 对真实缺口更诚实了还是更会假装覆盖？

如果一套测试最后主要只能回答：

- “第 7 个提示词现在有没有选中我心里想的那个 skill？”

那它就不是一个合格的正式框架，它只是在把项目重新拖回场景 patch。

## 最终立场

下一代 `vibe` 路由系统，应当被当成一个“分层证据检索加组合决策系统”来测试，而不是一个单一黑箱打分器。

因此，这份文章正式固定以下测试合同：

- `SKILL.md` 是唯一事实来源；
- 测试对象是路由结构，不是提示词传说；
- 默认回归集由合成语料和结构性缝面构成；
- 真实本地语料审计统一降级为可选审计；
- 成功与否由召回、泄漏、覆盖、风格稳定性、合同服从度和诚实缺口报告共同判定。

这就是当前阶段最适合作为正式方案的测试框架。它保护的是通用路由能力，而不是某几道题的临时表象。
